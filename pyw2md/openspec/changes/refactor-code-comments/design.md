## Context
项目当前存在严重的注释质量问题：
- 过度注释显而易见的代码（如简单getter方法的docstring）
- 中英文混杂的注释风格
- 冗长的文档字符串重复函数名信息
- 用注释弥补糟糕的代码可读性

## Goals / Non-Goals
**Goals:**
- 建立简洁实用的注释规范
- 删除所有冗余和无用注释
- 提升代码自解释能力
- 统一注释风格为简体中文

**Non-Goals:**
- 完全移除所有注释（保留有价值的"为什么"注释）
- 改变代码功能逻辑
- 修改API接口

## Decisions
- **决策**: 采用Linus Torvalds的实用主义注释哲学
- **理由**: 代码应该是最好的文档，注释应该解释"为什么"而不是"是什么"
- **替代方案**: 考虑过详细文档化，但会增加维护负担且违背简洁原则

**决策**: 统一使用简体中文注释
- **理由**: 项目为中文项目，使用母语更自然直观
- **替代方案**: 英文注释，但不符合项目实际情况

## Risks / Trade-offs
- **风险**: 过度删除注释可能影响代码理解
- **缓解**: 保留解释复杂逻辑和决策原因的关键注释
- **权衡**: 简洁性 vs 完整性 - 倾向简洁，让代码自解释

## Migration Plan
1. **阶段1**: 清理垃圾注释（1-2小时）
   - 删除简单方法的docstring
   - 移除自解释代码的行内注释

2. **阶段2**: 统一风格（2-3小时）
   - 统一中文注释
   - 简化冗长文档

3. **阶段3**: 代码重构（3-4小时）
   - 优化命名
   - 重构复杂逻辑

4. **阶段4**: 验证更新（1小时）
   - 测试验证
   - 文档更新
